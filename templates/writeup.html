A Terribly Boring Blurb on a Terribly Boring Site About the Not-So-Terribly Boring Django Framework

What is Django?

Django is a Python based web framework built for speedy development and clean design. Originally developed with tight deadlines in mind, some on the order of a few hours from conceptualization to launch, Django allows web developers and designers to build and launch applications quickly and easily by

  • providing a rich set of frameworks (or “apps”, as they put it) to draw from,
  • abstracting away complex and tedious configuration tasks into 3 or 4 commands,
  • and encouraging simple, lean and reusable designs with generic views and an easy to use templating system.

The Design: MTV?

Like many other popular web frameworks, Django follows the traditional Model/View/Controller (MVC) design pattern, albeit with a few caveats. Each django app, when initialized, comes with (at least) two files, a models.py, which manages the data model and views.py, which manages (or controls) the views. The actual views take shape in a set of HTML templates, which can often be swapped in or out or otherwise edited as the designer pleases - allowing him or her to easily modify the appearance of a site, but not necessarily what is seen on the site.

The Philosophy: Reduce, Reuse, Recycle.

The templating system, along with the rich set of included frameworks and an easy way of deploying other third-party apps reduces redundancy - saving developers time when it comes to development and debugging. Django’s “Don’t Repeat Yourself” (DRY) attitude manifests itself in generic views and templates. “Hot-pluggable” apps and frameworks included in Django, like the admin site, or the easy to use comments framework provide functionality nearly right out of the box. Many third-party apps and frameworks can also be easily added and deployed, capitalizing on a large number of Django packages shared on the internet.

The Potential: It’s fast... and it scales.

Django allows developers and designers to hit the ground running without having to worry much about confusing database or web server configurations - instead allowing them to work with an integrated web server and SQLite databases. When the time comes to deploy your site into a production environment, all you need to do is reconfigure a few settings and following a few easy instructions. When your site goes big, and you need to scale, Django’s architecture lets you, all you have to do is add more hardware - the guys over at Disqus know exactly how good it is.

What Isn’t Django?

For each of those good things we’ve just said about Django, we’re sure that there’s some sort of criticism of it by a fanboy or fangirl of a competing web framework. As such, below we discuss some of the pros and cons of using Django as opposed to other, similar frameworks.

It’s not Rails

Rails, or more formally, Ruby on Rails (RoR), is a popular web development framework based in Ruby. Like Django, Rails follows the MVC design pattern and focuses on minimizing code by adhering to the DRY principle. Also like Django, Rails makes use of an integrated development web server and SQLite databases, allowing developers to move quickly without worrying about the muddy details of deployment. For the most part, from a high level, the two frameworks are very similar, but as you dive deeper, core differences emerge.

Rails and Django differ most in their philosophies - which greatly impacts how they do things. Whereas RoR works with implicit assumptions in mind, emphasising Convention Over Configuration (CoC), Django goes in the other direction, instead favouring explicitness over implicitness. While Rails automagically seems to handle most configuration tasks out of sight of the developer, Django seems to want him or her to be directly involved in the process, though without making the process a burden.

It’s not TurboGears

TurboGears (version 2 specifically, TG2) is a Python based “rapid development megaframework” inspired by the likes of Django and Rails. Much like Rails and Django, TG2 follows the MVC design pattern. Unlike the two however, TurboGears is built on top of other third-party components like SQLAlchemy which handles the data model, Genshi which handles the views and Pylons which acts as the controller. All those third-party components aside, TurboGears lacks the "pluggable" nature that Django's apps have, with no real support for plugins.

Despite its decentralized structure, TurboGears tries to cater to new developers, it’s philosophy being “...focused on user experience, and creating a novice-friendly environment. We ship a smaller subset of components, and thus are better able to focus, test, and document things so that new users have the best possible experience.” However, whereas both Django and Rails have large user bases and a large wealth of documentation and examples to draw on, the TG2 community is small, and can’t boast big names like Rails can with Twitter or Django can with Rdio (that is, unless you don't count SourceForge)`.

It doesn’t do it all

No one single framework can do it all – no one single framework will attract everyone in droves – each has its own strengths and weaknesses. Django’s minimalist design forces users to be actively involved in their site’s configuration and deployment, but makes it easy. Rails’ “magic” takes care of a lot of the heavy lifting, producing results with minimal effort. TurboGears’ puts the developer in control of his or her application – it’s internal components able to be modified and worked on in ways that the other two cannot be (due to their highly integrated and dependent internal modules).
